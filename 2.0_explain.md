### AddressIntelligence.py：

### 1. 数据结构层 (MatchStatus & MatchResult)
*   **作用**：统一定义地址匹配的状态和返回结果
*   **解决的问题**：规范化模块输出，不仅输出提取出的省市区镇村（五级），还包含剩余的详细地址（`remainder`）和用于后续分流的**置信度打分**（逻辑得分、模糊得分、向量得分）。
*   结构体
---
### 2. 文本预处理器 (Preprocessor)
*   **作用**：清理用户输入的“脏”地址文本，统一行政区划的后缀命名。
*   **使用的方法**：
    *   全角转半角（通过字符映射表 `_FW_TABLE`）。
    *   正则替换（去除括号及内部内容，去除中英文字母数字以外的特殊符号）。
    *   别名统一（如把 `街道办事处` 替换为 `街道`，`村民委员会` 替换为 `村`）。
*   **解决的问题**：用户输入极其随意（带有括号说明、奇怪符号、或者使用了全称而非简称），导致字面量无法匹配。
*   **输入/输出示例**：
    *   **输入**：`“广东省 深圳市南山区粤海街道办事处１２号（原高新园区）”`
    *   **输出**：`“广东省深圳市南山区粤海街道12号”`
    *   
*   **注意**：去除括号及内部内容，可能存在部分信息遗漏

---
### 3. 地址路径生成工具 (`_build_path_keys`)
*   **作用**：将一行标准的五级地址记录，裂变成各种可能的用户**输入前缀路径**。
*   **使用的方法**：双层 `for` 循环遍历（生成不同起点和终点的组合），加上**跳级拼接逻辑**。
*   **解决的问题**：
    1.  **省略上级**：用户经常不写省市，直接写“南山区粤海街道”，双层循环允许从**省/市/县**分别作为起点构建索引键。
    2.  **跳跃层级**：用户可能跳过“街道”直接写“区+社区”（如：“永定区南庄坪永康社区”简写为“永定区永康社区”）。代码剥离了街道后缀，把核心名和村直接拼接，生成专门的“跳级路径”。
*   **输入/输出示例**：
    *   **输入**：一条记录 `{'province': '广东省', 'city': '深圳市', 'county': '南山区', 'township': '粤海街道', 'village': '科技园社区'}`
    *   **输出**：生成一组Key列表供自动机学习：
        *   `"广东省深圳市南山区..."`
        *   `"深圳市南山区..."`
        *   `"南山区粤海街道..."`
        *   `"南山区科技园社区"`（跳级路径）

---
### 4. 核心匹配引擎 (PathAhoCorasickIndex)
*   **作用**：负责**极速的字符串多模式匹配**，是Fast Path（快速通道）的核心。
*   **使用的方法**：`Aho-Corasick`（**AC自动机**）算法（借助 `pyahocorasick` 库）。这是一种多模匹配算法，可以在 $O(N)$（N为输入文本长度）的时间复杂度内，同时在文本中查找数百万个地址前缀。
*   **解决的问题**：如果用传统的正则或者数据库 `LIKE` 查询千万级的地址库，一次请求可能需要几秒钟。AC自动机将匹配耗时压缩到了微秒级。方法 `greedy_cover` 采用贪婪策略，优先选出最长的一条命中路径。
*   **输入/输出示例**：
    *   **输入（构建时）**：百万级的路径 Key（如上一步生成）。
    *   **输入（查询时）**：`"深圳市南山区粤海街道高新南十道"`
    *   **输出**：命中 `Entry(key="深圳市南山区粤海街道", level="township")`，剩余未匹配部分为 `"高新南十道"`。

*注： 时间复杂度：量化 算法运行时间与输入数据规模之间的**关系**，O(n) —— 线性时间
---

### 5. 主数据仓库 (MasterDataStore)
这是对前面所有组件进行统筹调度的“大总管”。

#### A. 索引加载阶段 (`load_excel`)
*   **作用**：读取本地Excel地址库，构建两个核心检索结构：**AC索引** 和 **分层下钻树 (`_drill_tree`)**。
*   **解决的问题**：不仅建立了高速自动机，代码还会过滤掉统计局数据中诸如“市辖区”、“直辖县级”等毫无意义的**虚假占位节点**。
*   同时构建的“下钻树”按地理层级把每个区县下面的**乡镇/村落核心名建立倒排索引**，用于后续兜底。

#### B. 核心匹配逻辑阶段 (`strict_match`)
这是该组件最重要的方法，采用了一种**多阶段渐进式匹配（瀑布流）**的设计：

*   **阶段一：AC 路径硬匹配 (Fast Path)**
    *   **动作**：直接用 AC自动机 在用户文本里扫一圈。
    *   **效果**：如果用户地址很标准，比如直接命中到“街道”或“村”级别（代码中定义为 `>= 3` 即 township级别以上），直接返回满分（1.0）通过。
    *   *   `0`: province (省)
*   `1`: city (市)
*   `2`: county (区/县)
*   `3`: township (乡/镇/街道)   **<-- 这就是 _STRICT_MIN = 3 的来源**
*   `4`: village (村/社区)

*   **错误：AC 路径硬匹配 (Fast Path) 必需精确到village**
*   **阶段二：分层下钻搜索 (Drill-down Fallback)**
    *   **触发条件**：AC自动机只匹配到了省、市或县（比如因为用户漏字、跳级或者遇到了被剔除的“市辖区”），断开了。
    *   **动作**：利用前面建立的 `_drill_tree`，在用户**剩下的字符串 (`remainder`)** 里，尝试去寻找该区县下挂载的镇/村的“核心名”（不带后缀的名字）。
    *   **解决的问题**：完美解决断层匹配。比如输入“南山区 科技园”，AC只匹配到南山区，剩下的“科技园”如果在下钻树里被发现属于“科技园社区”，系统会将它们拼在一起，并**逆向恢复**出中间省略的“粤海街道”。
    *   只探索**前确定节点**下挂载的乡镇和村落字典。
*   **阶段三：CPCA 辅助层 (L0.5 备用层)**
    *   **动作**：如果前两步还是没拿到满意的结果，调用第三方的 `cpca` 库（一个基于CRF/正则的开源地址提取库）作为规则库的补充，提取出省市区。得分设为 0.6分。
*   **阶段四/五：L2/AI 兜底钩子 (`ai_fallback_hook` / `l2_fallback_hook`)**
    *   **动作**：如果规则层彻底失败（得分低于阈值），触发留给下层（如 Faiss向量检索引擎 或 MGeo大模型）的扩展接口。

*   **输入/输出示例 (`strict_match` 的整体表现)**：
    *   **输入**：`"南山区科技园科技南十二路9号"`
    *   **输出**：
        ```python
        MatchResult(
            status=MatchStatus.STRICT_MATCH,
            matched_text="广东省深圳市南山区粤海街道科技园社区", # 补全了省市和街道
            province="广东省", city="深圳市", county="南山区",
            township="粤海街道", village="科技园社区",
            remainder="科技南十二路9号",  # 成功分离出精准的门牌号
            logic_score=1.0, final_score=1.0
        )
        ```

### 总结
这套 L0 层代码通过 **“正则清洗 -> 构建前缀置换集合 -> AC自动机极速初筛 -> 核心名下钻树兜底修复”** 的组合拳，解决的是**如何在千万级标准地址库中，快速且容错地提取出非标准用户输入中的“行政区划部分”，并干净地剥离出“详细门牌号”**。它完全在CPU及内存中进行，不依赖外部数据库，是典型的重构型高性能计算模块。


### Extactor.py：

L1 层的核心设计原则是：**“极度保守、宁缺毋滥”——宁可不匹配交由人工，也绝不把 AI 猜错的“幽灵地址”混入系统中**。

L1 在这套系统里处理的是：**写法极不规范、L0 完全看不懂的地址**（「崇文办事处南正居委会」这类省了上级地名的简写）。这类地址在这个数据集里 < 0.1%，所以 L1 的价值本来就很小，这是正常的。

### 1. 动态评分机制 (`_calc_fuzzy_score`)
*   **作用**：给 AI 模型提取出来的地址要素打分（置信度），决定是否采信 AI 的结果。
*   **使用的方法**：多维度动态加权计算。
    1.  **基础分**：提取到的信息越细，分越高（省 +0.05，县/区 +0.10，镇/街道 +0.12）。因为只提到“省”的废话没用，提到“街道”才有定位价值。
    2.  **连续性加分**：如果 AI 提取的层级是连贯的（比如连续提出了省、市、区），给予连续性加分。
    3.  **POI 惩罚（正则特征抑制）**：如果原始文本里包含了“大厦、广场、小区、x栋x室”等特征（使用正则 `_POI_PATTERN` 扫描），直接倒扣 0.15 分。
*   **解决的问题**：
    *   **防误判**：AI 很容易把“西湖公园”里的“西湖”当成“西湖区”。有了 POI 惩罚，只要文本里带有“公园”，置信度就会暴跌，从而阻止系统将它错误升级为行政区。
    *   **永远不给满分**：L1 层的得分上限被死死锁在 `0.85`（满分 1.0），以此告诉下游：**“这是 AI 猜的，不保证 100% 对”**。
*   **输入/输出示例**：
    *   **输入**：AI 提取出 `{'province': '广东', 'county': '南山区'}`，原文本为 `"广东南山区科技园"`。
    *   **输出**：得分可能为 `0.45`（触发了“园”字的 POI 惩罚，得分极低，后续会被系统抛弃）。
*   **评分策略（动态，基于提取层级完整度）**：
    fuzzy_score 由 _calc_fuzzy_score() 动态计算，而非固定 0.85：
      - 每个提取字段 +0.08 基础分（上限 0.40）
      - 从省向下连续不断层 ≥3 级，+0.10 连续性加分
      - 存在 POI 特征（楼栋号/大厦/广场），-0.15 惩罚
      - 上界 0.85，AI 层永远不触及 L0 的满分 1.0

*   **status 赋值规则（保守）**：
     - fuzzy_score >= 0.75 且通过标准库验证 → PARTIAL_MATCH（不升级 STRICT）
     - 0.55 <= fuzzy_score < 0.75          → PARTIAL_MATCH，写 active_buffer
     - fuzzy_score < 0.55                  → AI_REQUIRED，人工处理
     - 注：L1 永远不赋予 STRICT_MATCH，STRICT 只由 L0 的规则逻辑授予。

### 2. 核心 AI 推理引擎 (`MGeoExtractor`)
*   **作用**：加载并运行真实的深度学习大模型（达摩院的 MGeo 模型），对地址进行 NER（命名实体识别）。
*   **使用的方法**：
    *   **模型调用**：使用 `modelscope` 的 `pipeline` 加载预训练模型。
    *   **极限压榨 CPU 性能**：强制绑定 CPU（不依赖昂贵的显卡），通过 `torch.set_num_threads(8)` 限制物理核数避免线程打架；通过 `torch.no_grad()` 关闭梯度计算，节省 30%-50% 的内存；遇到批量任务 (`batch_extract`) 时打包矩阵，触发 CPU AVX 向量化加速。
    *   **防御性锁**：用 `_threading.Lock()` 把推理过程锁住，避免高并发下模型崩溃。
*   **解决的问题**：L0 层是基于“词典”的，如果词典里没有或者用户写了极其生僻的缩写，L0 会瞎。此模块通过“结合上下文猜词性”来提取省市区镇村。同时解决了 Python 跑深度学习模型容易内存溢出和卡死的问题（加了 128 字符截断限制）。
*   **输入/输出示例**：
    *   **输入**：`"深圳市张江高科园区66号"`（一个杂糅的地址）
    *   **输出**：模型输出 `[{'type': 'city', 'span': '深圳市'}, {'type': 'town', 'span': '张江'}]`，被解析封装为 L0 同款的 `MatchResult` 对象返回。
*   **实际使用识别率很低**：MGeo 本质是 NER 标注模型，只做「这段文字是什么类型的地名」，不做「这条路属于哪个社区管辖」的地理推断——后者需要地图数据库支持。

### 3. 柔性安全集成层 (`AIEnhancedStore`)
*   **作用**：这是 L1 和 L0 的“连接枢纽”。它继承了 L0 的类，并覆写了 `ai_fallback_hook` 钩子。负责对 AI 给出的结果进行**严苛的审计和合并**。
*   **使用的方法**：
    1.  **门控阈值（Gate Control）**：如果上一步算出的 `fuzzy_score < 0.55`，直接将 AI 结果扔进垃圾桶，状态改为 `AI_REQUIRED`（需要人工介入）。
    2.  **跨区域一致性校验 (`_is_consistent`)**：**这是 L1 层最核心的安全底线**。它拿着 AI 猜出来的“镇/村”，去 L0 构建好的“辖区下钻树 (`_drill_tree`)”里查岗。
*   **解决的问题**：防止生成**“缝合怪/幽灵地址”**。
    *   AI 模型不懂地理常识。如果 L0 已经确定了这是“广东省-深圳市-南山区”，但由于文本里写了“天河大厦”，AI 可能会自作聪明提取出一个“天河区”（位于广州）。
    *   `_is_consistent` 方法会发现：“天河”这个词，根本不在“深圳南山区”的合法街道/村落字典里！于是**果断拒绝合并**，以此保证数据的绝对纯洁。
*   **输入/输出示例（整体运转过程）**：
    *   **前情提要**：用户输入 `"湖南张家界慈利南山路9号"`。L0 提取到了 `{'province': '湖南省', 'city': '张家界市', 'county': '慈利县'}`，但遇到了断层，得分只有 0.6，传给了 L1 层。
    *   **L1 介入**：MGeo 模型介入，从 `南山路` 中提取到了 `township="南山街道"`，给了个及格分 `0.65`。
    *   **查岗校验**：`AIEnhancedStore` 拿着 `南山街道` 去 `_drill_tree` 里的 `('湖南省', '张家界市', '慈利县')` 下面找。结果发现慈利县根本没有叫“南山”的街道（深圳才有）。
    *   **最终输出**：校验失败，拒绝 AI 提取的街道。保留 L0 的结果，打上 `MatchStatus.PARTIAL_MATCH` 和 `needs_review=True` 标签，输出进待人工复核的缓冲池中。

### 总结
L1 层不仅仅是一个“套壳调用大模型”的代码，它是一个**带有重重安检门的漏斗**。
它巧妙地利用了 L0 层的数据字典 (`_drill_tree`) 来约束和校验大模型的“幻觉”，做到了**“让 AI 发挥语义理解的优势，同时用传统规则死死卡住 AI 的安全底线”**。最高只给 `PARTIAL_MATCH` 的设定，确保了高置信度的自动化流程绝对不会被 AI 污染。

**L2层（向量语义召回引擎 / JudgeEngine）**

 **L2 就是一个带有限制器（安全锁）的“全库模糊搜寻雷达”**。当用户输入的地址错别字太多、大量使用非标准简称（如“经开区”对应“经济开发区”），导致 L0 截断、L1 拒收时，L2 会把整段文本转化为**高维数学向量**，去标准库里寻找“语义最相近”的地址。

为了确保金融/政务环境的绝对安全，L2 层秉持 **“Fail-Closed（默认拒绝）”** 和 **“Structure First（结构优先）”** 原则，绝不让向量库瞎猜。

以下是详细的工作流、方法拆解与数据示例：

---

### 一、 L2层解决了什么问题？采用了什么方法？

#### 1. 解决的核心问题
*   **非标准简称与重度错别字**：比如标准库是“经济开发区”，用户输入“经开区”；标准库是“高新技术产业园”，用户输入“高新园”。L0的自动机匹配不上，L1能提取出实体但无法映射到标准库。
*   **打破字面限制**：向量检索比较的是“意思/语义”，而不是“字面量”，能够实现强大的模糊容错。

#### 2. 采用的核心方法
*   **Transformer 语义向量化**：使用 `bge-small-zh-v1.5` 模型（通过 mean-pooling 和 L2归一化），把每一条地址变成一个 384 维的浮点数向量（代表地址的特征）。
*   **Faiss 高速检索**：构建 `IndexFlatIP`（内积索引）。由于向量做了 L2 归一化，内积直接等同于“余弦相似度（Cosine Similarity）”，实现了毫秒级的千万级全库比对。
*   **Sigmoid 温度校准（惩罚机制）**：中文地址经常因为“省市区”几个字一样，导致余弦相似度虚高（动不动就 0.9 以上）。代码中用 `sigmoid((cosine - 0.75) * 8)` 把分数狠狠压低（0.9的相似度会被压到0.77），防止高估。
*   **物理隔离与结构约束 (`_structure_consistent`)**：**这是 L2 的灵魂**。不管向量库觉得多相似，找出来的候选地址**必须**包含 L0 已经确认的“省/市”。

---

### 二、 L2 核心工作流 (Workflow)

L2 层分为**离线建库**和**在线推理**两条流：

#### 1. 离线建库工作流 (`build_index`)
1.  **加载标准库**：读取包含 3万+（或千万级）条标准地址的 Excel。
2.  **文本拼接**：把每行的五级地址拼成大白话，如 `"湖南省 张家界市 慈利县 零阳街道 白沙井社区"`。
3.  **计算向量**：用 Transformer 模型批量把这些文本变成 384 维的矩阵。
4.  **构建 Faiss 索引**：连同各地的“区县白名单”一起，打包存入磁盘 `judge.index`。

#### 2. 在线兜底工作流 (`enhance`)
当且仅当 L0 止步于县级（PARTIAL）或 L1 拒绝（AI_REQUIRED）时触发：
1.  **触发与组装**：拿 L0 提取的残骸（如已识别的省市 + 剩下的门牌号）拼成 `query` 文本。
2.  **向量搜寻 (Semantic Search)**：将 `query` 转为向量，去 Faiss 库里捞取最相似的 **Top-10** 候选地址。
3.  **结构过滤 (Structure First)**：拿 L0 已经确定的省市，去卡这10个候选。省市不一致的，**直接删掉**。
4.  **补全检查 (Check Filled)**：如果存活的候选地址，不能给当前结果补充任何新的信息（比如没补全县、镇、村），则**放弃补全**。
5.  **字段合并与降权 (Merge & Discount)**：提取存活第一名的镇/村补进去，并给向量分打 8 折（`vector * 0.8`），作为最终得分。
6.  **输出结果**：如果得分及格，状态升级为 `PARTIAL_MATCH` 并打上 `needs_review` 标签，否则退回 `AI_REQUIRED`。

### 三、 数据示例 (Data Examples)

为了让您直观理解，我们来看两个典型场景：

#### 场景 1：完美兜底（非标准简称补全）

*   **用户输入**：`"湖南省 张家界市 经开区C区"`
*   **L0 层状态**：只认出了省市，遇到“经开区”卡住了。
    *   输出：`province='湖南省'`, `city='张家界市'`, `remainder='经开区C区'`, status=`PARTIAL_MATCH` (得分0.5)。
*   **进入 L2 层**：
    *   **向量搜索**：拿 `"湖南省张家界市经开区C区"` 去搜 Faiss。
    *   **Faiss 捞回的第一名**：标准库记录 `{'province':'湖南省', 'city':'张家界市', 'county':'慈利县', 'township':'经济开发区'}`（余弦相似度 0.92）。
    *   **结构校验**：L0的省市(湖南/张家界) == 候选的省市(湖南/张家界)。**校验通过！**
    *   **补全检查**：候选地址提供了 `county='慈利县'` 和 `township='经济开发区'`，成功填补了 L0 的空白。
*   **L2 输出**：
    ```python
    MatchResult(
        status=MatchStatus.PARTIAL_MATCH, 
        province="湖南省", city="张家界市", county="慈利县", township="经济开发区",
        remainder="C区", # 成功把门牌号剥离
        vector_score=0.79, # sigmoid(0.92)
        meta={'l2_suggest_strict': True, 'source': 'L2_JudgeEngine'}
    )
    ```
    *效果：成功将口语化的“经开区”映射回标准库的“慈利县经济开发区”。*

#### 场景 2：被“结构约束”拦截的误杀（Fail-closed 拦截）

*   **用户输入**：`"广东省 深圳市 南山村 8栋"`
*   **L0 层状态**：认出了广东和深圳。因为深圳南山区没有叫“南山村”的，L0停在市级。
    *   输出：`province='广东省'`, `city='深圳市'`, `remainder='南山村 8栋'`
*   **进入 L2 层**：
    *   **向量搜索**：拿 `"广东省深圳市南山村8栋"` 去搜。
    *   **Faiss 捞回的第一名**：标准库记录 `{'province':'福建省', 'city':'泉州市', 'county':'惠安县', 'village':'南山村'}`（因为“南山村”三个字完全一致，导致余弦相似度极高）。
    *   **结构校验**：L0要求必须是 `广东省 深圳市`。候选地址是 `福建省 泉州市`。**校验彻底失败！**
    *   **拦截动作**：L2 触发 `filtered 为空` 逻辑。
*   **L2 输出**：
    ```python
    MatchResult(
        status=MatchStatus.AI_REQUIRED, # 拒绝兜底，丢给人工处理
        province="广东省", city="深圳市", remainder="南山村 8栋",
        meta={'l2_filtered_empty': True}
    )
    ```
    *效果：如果没这个机制，系统会把一个深圳的地址强行定位到福建去。L2 的结构约束完美阻止了这种灾难性幻觉。*

### 总结
L2 层的 `JudgeEngine` 把大模型的语义理解能力和传统数据结构的严谨性结合在了一起。它就像一个被上了枷锁的超人，**允许你进行发散联想（向量检索），但联想的结果必须在既定的物理框架（省市白名单）内**，以此在“提高召回率”和“保证准确率”之间做到了极致的平衡。

## 三个疑问的完整回答

### 疑问1：village 为空——两个不同的问题

**不是「L0 只匹配到 township 级」导致的**，而是两种情况混在一起：

**情况A1（531条，68%）**：地址本身只写到路名/门牌/楼栋，如「合口镇迎宾中路055号」，里面根本没有村级信息。无论 L0 算法多强都无解，这是数据采集端的上限。

**情况A2（247条，32%）**：地址里写了「群丰**村**」「樟木**村**」，但标准库记录的是「群丰**社区**」「樟木**社区**」（农村改制后更名）。AC key 不匹配导致漏识别。**这是可修复的** — 在 `_build_path_keys` 里为每个 village 字段额外生成「村↔社区」互换的 alias key，覆盖全部 247 条。

---

### 疑问2：括号暴力删除——确实有损失，但影响量小

当前全库 9193 条里只有 5 条受影响，其中真正可以改善的是 1 条：「张家界经济开发区A区（永定区西溪坪办事处…）」，括号内藏着真实行政地址，删掉后 L0 和 L1 都失去了识别机会。

**推荐修复**：两段式解析——主体地址先跑 L0，若结果 level < township，则将括号内容拼接后再跑一次，取结果更好的那个。不改 Preprocessor 本身，在 `strict_match` 的上层调用处实现。

---

### 疑问3：L2 向量检索的根本性局限——你的判断是对的

数据验证确认这是真实风险：

- 全省有 **1542 个**村名跨 3+ 个县区，「和平村」存在于 45 个不同县
- 张家界市内有 **36 个**跨县重名村（「太坪村」同时属于永定区、桑植县、慈利县）
- `_structure_consistent` 的 city 约束只能拦截**跨市**歧义，同市内的重名完全无法区分
- 向量相似度对「同名不同县」的地址几乎没有区分能力（字符串一致时余弦差 < 0.01）

**当前 19 条 L2 补全的 township/village 正确性系统内部无法验证**，L2 实质上是在 county 未知时做有限的随机猜测。

**根本设计问题**：L2 把「候选生成」和「最终决策」合并在一起，当存在多个结构一致的候选时，直接取 Top1 是不科学的。

**短期可行的改进**：在 `judge_engine.py` 的 `enhance()` 里加歧义检测——

```python
# filtered 里有 2+ 条候选且 Top1/Top2 分差 < 0.05 时，强制 needs_review
if len(filtered) >= 2:
    score_gap = filtered[0][1] - filtered[1][1]
    if score_gap < 0.05:
        partial_result.meta['l2_ambiguous'] = True
        partial_result.meta['l2_candidates'] = [
            {lv: c.get(lv,'') for lv in _LEVELS}
            for c, _ in filtered
        ]
        # 不写入 township/village，保留人工判断
        partial_result.meta['needs_review'] = True
        partial_result.status = MatchStatus.AI_REQUIRED
        return partial_result
```

**长期建议**：L2 的职责应该改为「生成候选列表」，由下游的人工审核或规则引擎做最终决策，而不是直接写入五级字段作为落地数据。
## 目前系统能做出的最合理架构决策

先看月亮湾花园这个案例本身揭示的问题：

L2 选了「月亮**岛**社区」，理由是「月亮湾」和「月亮**岛**」都含「月亮」，余弦相似度被字面重叠拉高。但实际正确答案是「大庸桥社区」——和输入文本几乎没有字符重叠。**正确答案反而相似度最低**，这是向量检索对地名匹配的根本性失败场景，无法通过调参解决。
